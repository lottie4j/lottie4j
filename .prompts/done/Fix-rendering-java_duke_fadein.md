## Lottie part diagnosis
- Layer:    the only animated `op` (opacity) keyframe layer covering the full duration.
  Inspect `layers[*]` for `ks.o.a == 1` with two keyframes 0 → 100.
- Shape(s): the layer contains a **gradient fill** (`{"ty":"gf",
  "s":{"k":[86.63,167.07]}, "e":{"k":[141.35,167.07]}, "t":2, …}`). `t:2` ⇒ radial
  gradient. Gradient stops live in `g.k.k` (color stops + alpha stops, ordered
  `pos,r,g,b, … , pos,a, …`).
- Animated property: layer opacity `ks.o` over the fade-in interval.
- Easing: most likely bezier (`i`/`o` x/y arrays), not linear — failing at exactly one
  frame is the signature of a keyframe-boundary off-by-one.

## Root cause hypothesis
At the exact frame where the opacity keyframe segment ends (or the layer's `ip` begins),
JavaFX renders one frame at the **wrong opacity** (probably 0 instead of the first
in-bound value, or 100 instead of the last). The single-frame dip from steady state
into 78 % is a classic edge-of-segment interpolation bug, not a gradient stop bug —
otherwise R/G/B would diverge for many frames. Likely culprits:

1. `com.lottie4j.core.helper` (Keyframe deserializer / `KeyframeInterpolator`) returning
   the *next* segment's start value instead of the *current* segment's end value at
   `t == keyframe.t`.
2. `com.lottie4j.fxplayer.renderer.layer.TransformApplier` — the opacity it computes
   for the boundary frame snaps to 0 (or 100) before `ip` is honoured.
3. `LottiePlayer.seekToFrame` (call-site in `CompareFxViewWithWebViewTest.waitForRender`)
   could be inclusive vs exclusive on the FX side and exclusive vs inclusive on the
   WebView reference.

## Proposed fix
- `lottie4j/core/src/main/java/com/lottie4j/core/model/keyframe/` — confirm the segment
  search returns segment `[k_i, k_{i+1})` for `t ∈ [k_i.t, k_{i+1}.t)`. Add a unit test
  evaluating at exactly `k_{i+1}.t` and one tick before it.
- `lottie4j/fxplayer/src/main/java/com/lottie4j/fxplayer/renderer/layer/TransformApplier.java`
  — verify opacity (`o`) is clamped only after interpolation, not before.
- If the gradient itself is the cause (R *and* G *and* B all drop by ~20 pp at the same
  frame, vs only opacity-driven uniform drop), look at
  `lottie4j/fxplayer/src/main/java/com/lottie4j/fxplayer/renderer/style/FillRenderer.java`
  for radial-gradient stop parsing (`g.k.k` raw array → JavaFX `Stop` list).
